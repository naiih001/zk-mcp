# Automated Releases Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add automated releases with `release-please`, a generated changelog, and an MIT license so versioning and release notes are driven from merged conventional commits.

**Architecture:** Keep CI responsible for validation and branch merging, and add a separate release workflow that only runs release orchestration. `release-please` will own the release PR, changelog generation, and version bumps in `package.json` and `package-lock.json`; the repo will carry the MIT license and release metadata needed for those outputs to be authoritative.

**Tech Stack:** GitHub Actions, `release-please-action`, npm, conventional commits, Markdown.

## Global Constraints

- Release automation must be driven by conventional commits.
- CI validation stays separate from release publishing.
- `package-lock.json` must stay in sync with release version bumps.
- Add or update `LICENSE` and `CHANGELOG.md`.

---

### Task 1: Add release-please configuration and workflow

**Files:**
- Create: `.github/release-please-config.json`
- Create: `.github/workflows/release-please.yml`
- Modify: `.github/workflows/ci.yml`

**Interfaces:**
- Consumes: GitHub Actions `contents: write` permissions, `main` branch merges, conventional commit messages.
- Produces: release PRs, generated changelog entries, version bumps, Git tags, and GitHub Releases.

- [ ] **Step 1: Add the release configuration file**

```json
{
  "packages": {
    ".": {
      "release-type": "node",
      "package-name": "zk-mcp"
    }
  },
  "changelog-sections": [
    { "type": "feat", "section": "Features" },
    { "type": "fix", "section": "Fixes" },
    { "type": "perf", "section": "Performance" },
    { "type": "refactor", "section": "Refactors" },
    { "type": "docs", "section": "Documentation" },
    { "type": "test", "section": "Tests" },
    { "type": "build", "section": "Build System" },
    { "type": "ci", "section": "CI" },
    { "type": "chore", "section": "Chores" }
  ]
}
```

- [ ] **Step 2: Add the release workflow**

```yaml
name: Release Please

on:
  push:
    branches:
      - main

permissions:
  contents: write
  pull-requests: write

jobs:
  release-please:
    runs-on: ubuntu-latest
    steps:
      - uses: googleapis/release-please-action@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
          config-file: .github/release-please-config.json
          manifest-file: .github/release-please-manifest.json
```

- [ ] **Step 3: Keep the existing CI workflow focused on validation**

Remove any release responsibilities from `.github/workflows/ci.yml` so it only tests and merges branches into `main`.

- [ ] **Step 4: Verify the YAML and JSON are structurally valid**

Run:
```bash
node -e "JSON.parse(require('fs').readFileSync('.github/release-please-config.json', 'utf8')); console.log('ok')"
```
Expected: `ok`
